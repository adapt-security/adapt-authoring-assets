# Assets

The `adapt-authoring-assets` module is the REST API and storage layer for media (images, video, audio, documents, fonts, subtitles, etc.). It extends `AbstractApiModule`, so it inherits the standard CRUD endpoints (`GET`/`POST`/`PUT`/`DELETE` on `/api/assets`) and adds a dedicated file-serving endpoint. Assets are stored on disk via a pluggable repository abstraction; image/video assets additionally get an ffmpeg-generated thumbnail and ffprobe-extracted metadata.

It registers with the `authored` and `tags` modules, so every asset also carries authored fields (created/updated by, timestamps) and supports tagging.

## Asset model

The stored document is defined in `schema/asset.schema.json`. Author-editable fields: `title` and `description` (both required, both searchable), and `url` (for external assets). Everything else is derived at upload time and read-only:

| Field | Type | Notes |
|---|---|---|
| `title` | string | required, searchable |
| `description` | string | required, searchable |
| `url` | string (uri) | only for external assets |
| `type` | string | `image`, `video`, `audio`, or other (the MIME type's first segment) |
| `subtype` | string | the MIME subtype, e.g. `png`, `mp4`, `svg+xml` |
| `size` | number | bytes |
| `hash` | string | SHA-256 of the file contents, indexed, used for duplicate detection |
| `path` | string | relative path to the stored file (internal) |
| `repo` | string | repository/storage type, defaults to `local` (internal) |
| `hasThumb` | boolean | whether a thumbnail was generated |
| `resolution` | string | e.g. `1920x1080`, from ffprobe (image/video) |
| `duration` | number | seconds, from ffprobe (video only) |

A unique-ish index is set on `hash` in `init()` (`mongodb.setIndex(this.collectionName, 'hash')`).

## Endpoints

Standard `AbstractApiModule` CRUD lives under `/api/assets`. The module adds one custom route (`routes.json`, root `assets`):

```
GET /api/assets/serve/:_id          # stream the asset file
GET /api/assets/serve/:_id?thumb=true   # stream its thumbnail
```

`serveAssetHandler` looks up the asset, builds a filesystem wrapper, and pipes a read stream to the response with `Content-Type: <type>/<subtype>`. If `thumb=true` is requested but the asset has no thumbnail (e.g. an SVG), it falls back to serving the main file.

Access is gated by `read:${scope}` (the standard authored/api scope for the assets collection).

`deleteMany` is explicitly disabled (`FUNC_DISABLED`) — assets must be deleted one at a time so their files and thumbnails are cleaned up.

## Upload

Uploads ride on the normal create/update requests. The `onRequest` hook (tapped onto `requestHook`) runs for modifying, non-DELETE requests and parses the incoming file before it reaches the insert/update handlers:

- `middleware.fileUploadParser(fileTypes, opts)` — multipart file upload
- `middleware.urlUploadParser(fileTypes, opts)` — fetch an asset from a remote URL

Both are passed `expectedFileTypes` (the MIME allow-list) and `maxFileSize`. A parsed file lands on `req.fileUpload.files.file[0]` and is merged into `req.apiData.data.file`. A comma-separated `tags` string in the body is split into an array.

On `insert`:
1. `checkDuplicate(filepath, size)` — cheap size pre-check, then SHA-256 hash compare; throws `DUPLICATE_ASSET` (409) if an identical file already exists.
2. Insert the DB record, then `createFsWrapper(doc).updateFile(file, hash)` moves the file into place, generates the thumbnail/metadata, and the record is updated with the derived fields.
3. If any of that throws, the just-inserted record is deleted (rollback).

`update` behaves the same when a new `file` is present (excluding self from the duplicate check); without a file it's a plain metadata update.

## Storage / repositories

Storage is abstracted behind `AbstractAsset` (`lib/AbstractAsset.js`). The only built-in implementation is `LocalAsset` (`lib/LocalAsset.js`), which stores files on the local filesystem under `assetDir` (default `$DATA/assets`). The stored filename is `<_id>.<ext>`, preserving the originally uploaded extension.

`AbstractAsset` defines the storage contract — `read`, `write`, `move`, `delete`, `ensureDir`, `ensureExists`, `resolvePath`, `assetRoot` — which subclasses override. `LocalAsset` implements these with Node `fs` streams.

Register additional repository types (e.g. S3) with `registerAssetType(MyAsset)`. The class must directly extend `AbstractAsset` and expose a unique static `name`; `defaultAssetRepository` (config, default `local`) selects which is used when an asset has no `repo`. `createFsWrapper(assetData)` instantiates the right repository from the asset's `repo` field.

On app ready, `performHousekeeping()` walks every asset, calls `ensureExists()` (logs missing files) and `generateThumb()` (back-fills missing thumbnails).

## Thumbnails (ffmpeg)

Thumbnails are generated by ffmpeg, bundled via `@ffmpeg-installer/ffmpeg`. `hasThumb` is set true for images (except SVG) and all video. `generateThumb()`:

- downloads the asset to a temp file (ffmpeg doesn't reliably handle streams),
- runs ffmpeg scaling to `thumbnailWidth` (default 320, height auto), grabbing 1 frame — for video, the frame at the 5-second mark (`-ss 00:00:05.000`),
- writes a `.png` (`thumbnailExt`) into `thumbnailDir` (default `$TEMP/asset-thumbs`).

The command run is roughly:

```
ffmpeg -i TEMP_<file> -vf scale=320:-1 -vframes 1 -update 1 [-ss 00:00:05.000] \
  -hide_banner -loglevel error <id>.png
```

Failures throw `FFMPEG` (500) with the command and cwd attached.

## Metadata (ffprobe)

For thumbnail-eligible assets, `generateMetadata()` runs ffprobe (`@ffprobe-installer/ffprobe`) over a temp copy with `-print_format json -show_format -show_streams`, then derives:

- `resolution` (`<width>x<height>`) from the video stream,
- `duration` (floored seconds) for video,
- `size` from the format block.

Failures throw `FFPROBE` (500).

### Custom binaries

If the bundled binaries don't suit your platform, override them in config:

```jsonc
// conf/<env>.config.js → "adapt-authoring-assets"
{
  "customFfmpegCommand": "/usr/local/bin/ffmpeg",
  "customFfprobeCommand": "/usr/local/bin/ffprobe"
}
```

When set, the module skips its bundled-binary executable-permission check (`ensureLibPermissions`) for that command and uses yours instead.

## Supported types

Uploads are restricted to the `expectedFileTypes` MIME allow-list (config). Defaults include:

- **Images:** gif, jpeg, png, svg+xml, webp
- **Video:** mp4, ogg, webm
- **Audio:** mpeg, wav
- **Documents:** pdf, doc(x), xls(x), ppt(x), rtf, zip, html
- **Subtitles:** x-subrip (srt), vtt
- **Fonts:** woff, woff2

Anything outside `audio`/`image`/`video` is grouped as `other`; `find()` rewrites a `{ type: { $in: ['other'] } }` query into `{ type: { $nin: ['audio','image','video'] } }`.

## Configuration

From `conf/config.schema.json`:

| Key | Default | Purpose |
|---|---|---|
| `assetDir` | `$DATA/assets` | local storage root |
| `defaultAssetRepository` | `local` | repository for new assets |
| `expectedFileTypes` | (see above) | upload MIME allow-list |
| `assetMaxFileSize` | `50mb` | max upload size (mutable, public) |
| `thumbnailDir` | `$TEMP/asset-thumbs` | thumbnail storage |
| `thumbnailExt` | `.png` | thumbnail file extension |
| `thumbnailWidth` | `320` | thumbnail max width (px) |
| `customFfmpegCommand` | _(unset)_ | override bundled ffmpeg |
| `customFfprobeCommand` | _(unset)_ | override bundled ffprobe |

## How content references assets

Content documents (`adapt-authoring-content`) reference assets by their `_id`. The content module maintains an indexed `_assetIds` array on each content document, recomputed on insert/update from the document's asset-typed fields. This powers asset-usage queries — e.g. "how many courses use this asset" — without the assets module needing to know about content. See the content module's `_assetIds` handling (`ContentModule.computeAssetIds`) and the asset-usage endpoint for details.

The browser fetches the actual file via `GET /api/assets/serve/:_id` (and `?thumb=true` for the listing/preview thumbnail).
